77 A decorator for test functions or fixtures, to parametrize them based on test cases. It works similarly to 

78 `@pytest.mark.parametrize`: argnames represent a coma-separated string of arguments to inject in the decorated 

79 test function or fixture. The argument values (argvalues in `pytest.mark.parametrize`) are collected from the 

80 various case functions found according to `cases`, and injected as lazy values so that the case functions are called 

81 just before the test or fixture is executed. 

82 

83 By default (`cases=AUTO`) the list of test cases is automatically drawn from the python module file named 

84 `test_<name>_cases.py` or if not found, `cases_<name>.py`, where `test_<name>` is the current module name. 

85 

86 Finally, the `cases` argument also accepts an explicit case function, cases-containing class, module or module name; 

87 or a list of such elements. Note that both absolute and relative module names are suported. 

88 

89 Note that `@parametrize_with_cases` collection and parameter creation steps are strictly equivalent to 

90 `get_all_cases` + `get_parametrize_args`. This can be handy for debugging purposes. 

91 

92 ```python 

93 # Collect all cases 

94 cases_funs = get_all_cases(f, cases=cases, prefix=prefix, glob=glob, has_tag=has_tag, filter=filter) 

95 

96 # Transform the various functions found 

97 argvalues = get_parametrize_args(host_class_or_module, cases_funs, debug=False) 

98 ``` 

99 

100 :param argnames: same than in @pytest.mark.parametrize 

101 :param cases: a case function, a class containing cases, a module object or a module name string (relative module 

102 names accepted). Or a list of such items. You may use `THIS_MODULE` or `'.'` to include current module. 

103 `AUTO` (default) means that the module named `test_<name>_cases.py` or if not found, `case_<name>.py`, will be 

104 loaded, where `test_<name>.py` is the module file of the decorated function. When a module is listed, all of 

105 its functions matching the `prefix`, `filter` and `has_tag` are selected, including those functions nested in 

106 classes following naming pattern `*Case*`. Nested subclasses are taken into account, as long as they follow the 

107 `*Case*` naming pattern. When classes are explicitly provided in the list, they can have any name and do not 

108 need to follow this `*Case*` pattern. 

109 :param prefix: the prefix for case functions. Default is 'case_' but you might wish to use different prefixes to 

110 denote different kind of cases, for example 'data_', 'algo_', 'user_', etc. 

111 :param glob: an optional glob-like pattern for case ids, for example "*_success" or "*_failure". Note that this 

112 is applied on the case id, and therefore if it is customized through `@case(id=...)` it should be taken into 

113 account. 

114 :param has_tag: a single tag or a tuple, set, list of tags that should be matched by the ones set with the `@case` 

115 decorator on the case function(s) to be selected. 

116 :param filter: a callable receiving the case function and returning `True` or a truth value in case the function 

117 needs to be selected. 

118 :param ids: optional custom ids, similar to the one in `pytest.mark.parametrize`. Users may either provide an 

119 iterable of string ids, or a callable. If a callable is provided it will receive the case functions. Users 

120 may wish to use `get_case_id` or other functions in the API to inspect the case functions. 

121 :param idstyle: This is mostly for debug. Style of ids to be used in the "union" fixtures generated by 

122 `@parametrize` if some cases are transformed into fixtures behind the scenes. `idstyle` possible values are 

123 'compact', 'explicit' or None/'nostyle' (default), or a callable. `idstyle` has no effect if no cases are 

124 transformed into fixtures. As opposed to `ids`, a callable provided here will receive a `ParamAlternative` 

125 object indicating which generated fixture should be used. See `@parametrize` for details. 

126 :param scope: the scope of the union fixture to create if `fixture_ref`s are found in the argvalues 

127 :param import_fixtures: experimental feature. Turn this to True in order to automatically import all fixtures 

128 defined in the cases module into the current module. 

129 :param debug: a boolean flag to debug what happens behind the scenes 

130 :return:

307 """ 

308 Transforms a list of cases (obtained from `get_all_cases`) into a list of argvalues for `@parametrize`. 

309 Each case function `case_fun` is transformed into one or several `lazy_value`(s) or a `fixture_ref`:

323 :param debug: a boolean flag, turn it to True to print debug messages. 

324 :return: 

325 """ 

326 return [c for _f in cases_funs for c in case_to_argvalues(host_class_or_module, _f, prefix, scope, import_fixtures,
